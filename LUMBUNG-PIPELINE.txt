##################################################
BAGIAN A — PETA BESAR: APA ITU LUMBUNG & BUAT APA
##################################################

KONTEKS FINAL: Pemerintah kabupaten ingin sistem dari Koperasi Padiwangi
direplikasi ke 4 koperasi desa lain yang berbeda karakter. Masalah lama:
(1) data bercampur antar-koperasi, (2) pengurus tak paham laporan,
(3) bingung berbagi data ke mitra pembiayaan.

STUDY CASE FINAL — 5 SKENARIO YANG HARUS DIJAWAB:
  A. Anggota (Pak Hendra) lintas koperasi ajukan pembiayaan
  B. 3 koperasi pengadaan pupuk bersama
  C. Penerimaan panen di gudang cold storage (dijawab di slide)
  D. Anomali transaksi kasir after-hours
  E. Pengawas eksternal audit perubahan data pinjaman

SOLUSI: LUMBUNG = "sistem operasi koperasi" — satu platform, banyak
koperasi, masing-masing terisolasi tapi bisa berkolaborasi dengan
kontrol dan audit penuh.

SISTEM — 8 LAYER:

1. LUMBUNG CORE (multi-tenant)
   APA  : satu platform untuk banyak koperasi; data tiap koperasi terpisah.
   BUAT : data Koperasi A tidak bocor ke Koperasi B.
   CARA : setiap baris data diberi koperasi_id; RLS di database memastikan
          user hanya bisa akses data koperasinya sendiri.

2. LUMBUNG SYNC (offline-first)
   APA  : aplikasi tetap jalan walau tidak ada internet.
   BUAT : Harapan Baru lokasi terpencil, sinyal susah; pengurus tetap bisa
          mencatat, data terkirim otomatis saat sinyal kembali.
   CARA : simpan ke Dexie (IndexedDB) dulu + antrian outbox -> kirim ke
          server saat event 'online'.

3. LUMBUNG PASS (paspor data) — HERO FITUR — jawab Case A
   APA  : cara aman berbagi data ke pemodal/mitra tanpa kirim buku besar.
   BUAT : jawab "bingung kirim data apa" + "Pak Hendra lintas koperasi".
          Pemodal verifikasi tanpa akses DB koperasi langsung.
   CARA : pilih data agregat -> consent per kategori -> hash SHA-256 ->
          link/QR time-limited -> audit log siapa buka kapan.

4. LUMBUNG LENS (laporan adaptif)
   APA  : dashboard + laporan mudah dipahami pengurus desa.
   BUAT : jawab "pengurus tak paham laporan". Angka jadi grafik + narasi.
   CARA : Recharts (bar/donut/garis) + kalimat otomatis (template/Haiku).

5. LUMBUNG INSIGHT (AI pendukung)
   APA  : kecerdasan yang mengubah data jadi sinyal terpercaya.
   BUAT : verifikasi ternak dari foto (anti fiktif) + skor kredit transparan.
   CARA : COCO-SSD on-device (hitung sapi) + scoring formula eksplisit.
   CATATAN: AI hanya MEMBANTU; keputusan tetap pengurus (aturan lomba).

6. LUMBUNG ATLAS (pemerintah kabupaten)
   APA  : dashboard ringkasan semua koperasi untuk pemkab.
   BUAT : pemkab mengawasi tanpa melanggar privasi anggota.
   CARA : tampilkan angka AGREGAT saja (VIEW database), bukan data mentah.

7. LUMBUNG GUARD (integritas & audit) — jawab Case D + E
   APA  : sistem deteksi anomali + audit trail untuk investigasi.
   BUAT : deteksi kasir curang (D), fasilitasi pengawas eksternal (E).
   CARA : trigger audit_log setiap INSERT/UPDATE/DELETE + query pola anomali
          + role pengawas (baca log saja, tidak akses PII anggota lain).

8. LUMBUNG PASAR (pengadaan bersama) — jawab Case B
   APA  : modul untuk koperasi berkolaborasi beli barang bersama.
   BUAT : 3 koperasi gabungkan kebutuhan pupuk/pakan -> harga lebih baik.
   CARA : tabel pengadaan + alokasi per koperasi + status rekening terhubung.

CASE C (Gudang cold storage): dijawab di PPT — arsitektur inventory +
jadwal offtaker ada di roadmap Q2. Konsep quality tracking ternak
sudah ada di modul Core (status sehat/pantau/sakit).

ALUR DEMO KILLER (3 menit):
  1. Catat ternak OFFLINE -> online SINKRON
  2. FOTO sapi -> AI hitung -> terverifikasi
  3. Terbitkan Lumbung Pass ke pemodal (Pak Hendra flow, Case A)
  4. Pemodal buka link -> Terverifikasi -> Setujui Pembiayaan
  5. Guard: tampilkan anomali kasir terdeteksi (Case D)


##################################################
BAGIAN B — GLOSARIUM TEKNOLOGI
##################################################

Next.js        : framework web (React) + PWA. Deploy gratis ke Vercel.
TypeScript     : JavaScript + tipe data -> lebih sedikit bug.
Tailwind/shadcn: styling & komponen UI siap pakai -> UI rapi cepat.
PWA            : web yang bisa diinstall seperti app & jalan offline.
                 Target HP Android murah, tanpa app store.
Supabase       : backend instan = PostgreSQL + Auth + API + Storage.
                 Hemat waktu besar untuk tim 2 orang.
RLS            : Row-Level Security — tiap user hanya akses data
                 koperasinya. Fondasi multi-tenant.
Dexie.js       : simpan data di browser (IndexedDB). Dasar offline-first.
Outbox         : antrian perubahan offline; dikirim otomatis saat online.
COCO-SSD       : model AI siap pakai, kenali objek "cow" di foto.
                 Jalan di browser, tanpa training ulang.
TensorFlow.js  : jalankan AI on-device di browser/HP.
SHA-256        : sidik jari digital data. Kalau data diubah 1 angka,
                 sidiknya berubah -> bukti data tidak dipalsukan (Pass).
audit_log      : tabel catat semua perubahan data: tabel apa, siapa,
                 kapan, nilai lama vs baru. Dasar Guard (Case D+E).
Recharts       : grafik bar/donut/garis untuk Lens.
Vercel         : hosting frontend gratis -> link publik untuk demo.
Claude Haiku   : LLM untuk narasi laporan otomatis. Opsional + fallback.

PHASE 7 — LUMBUNG INSIGHT (AI + Sistem Jaminan)
==================================================
Buat apa: verifikasi aset sebagai pengganti slip gaji (konteks desa),
skor kredit transparan, dan sistem limit bertahap. AI hanya membantu —
keputusan tetap di pengurus/pemodal.

[ ] 7.1  lib/vision.ts -> COCO-SSD verifikasi ternak via foto [paling nge-wow]
         Cara kerja: pengurus buka kamera HP di app (browser camera API)
         -> foto ternak -> TF.js load COCO-SSD di browser HP itu sendiri
         -> detect class "cow" -> hitung berapa yang terlihat
         -> simpan jumlah + flag terverifikasi di data ternak
         Fallback: input manual kalau model gagal (tetap functional).
         -> Buat apa: cegah "ternak fiktif" saat jadi agunan ke pemodal.
         Di desa, orang tidak punya slip gaji tapi punya ternak nyata.
         COCO-SSD membuktikan ternak itu ada secara objektif — lebih
         susah dipalsukan dari dokumen kertas. Tidak perlu hardware,
         cukup kamera HP yang sudah dimiliki pengurus.

[ ] 7.2  lib/scoring.ts -> skor kelayakan kredit TRANSPARAN (2 jalur)
         JALUR A — anggota dengan track record:
           skor = 0.4 * rasio_ternak_sehat
                + 0.4 * (1 - rasio_angsuran_macet)
                + 0.2 * pertumbuhan_simpanan_3bln
         JALUR B — first-time borrower (belum pernah pinjam):
           skor = 0.5 * nilai_aset_terverifikasi / nilai_aset_klaim
                + 0.3 * (ada_penjamin ? 1 : 0)
                + 0.2 * (durasi_anggota_bulan / 24)  -- maks 2 tahun
         UI: tampilkan skor + breakdown per komponen + jalur yang dipakai
         -> Buat apa: pemodal tahu dari mana angka skor berasal.
         First-time borrower tetap bisa dapat skor walau tanpa riwayat
         cicilan — dinilai dari aset nyata + kepercayaan komunitas.
         Pengurus tahu apa yang perlu diperbaiki untuk naikkan limit.

[ ] 7.3  Peringatan dini mortalitas ternak
         Kalau ternak status "mati" bulan ini > 10% total
         -> badge merah di beranda: "2 dari 8 ternak mati bulan ini"
         -> Buat apa: sinyal dini sebelum kerugian membesar. Juga
         otomatis flag Pass yang outstanding sebagai "aset berubah".

[ ] 7.4  Sistem jaminan aset lengkap (pengganti slip gaji)
         Tiga jenis aset yang bisa dijadikan jaminan:
         (a) TERNAK — diverifikasi COCO-SSD (objektif, susah dipalsukan)
             kolom: jumlah_klaim, jumlah_terverifikasi_ai, nilai_estimasi
         (b) LAHAN — foto + input manual (luas, status: milik/garap)
             kolom: foto_lahan_url, luas_m2, status_lahan, nilai_estimasi
         (c) ALAT PRODUKSI — foto + input manual (jenis, kondisi)

PHASE 8 — LUMBUNG GUARD (integritas & audit) — Case D + E
==================================================
Buat apa: deteksi kasir curang tanpa langsung menuduh (Case D) dan
fasilitasi pengawas eksternal audit data tanpa lihat PII anggota (Case E).
Satu tabel audit_log menyelesaikan keduanya sekaligus.

[x] 8.1  Supabase trigger di semua tabel utama (ternak, pakan, pinjaman,
         angsuran, simpanan):
         Setiap INSERT/UPDATE/DELETE -> catat ke audit_log:
           (tabel, row_id, aksi, kolom_diubah, nilai_lama JSONB,
            nilai_baru JSONB, dilakukan_oleh, dilakukan_pada)
         -> Buat apa: rekam jejak semua perubahan data secara otomatis.
         Pengurus tidak bisa "diam-diam" ubah data tanpa tercatat.
         Ini fondasi untuk Case D dan E sekaligus.

[x] 8.2  lib/anomaly.ts -> deteksi pola anomali kasir (Case D):
         Query: hitung transaksi void/cancel per user setelah jam 18:00
         Threshold: > 3 kali dalam 7 hari -> flag "Perlu Tinjauan"
         Tampil di beranda pengurus: badge merah + nama kasir +
         jumlah kejadian + tombol "Lihat Detail"
         PENTING: TANPA tuduhan langsung. Tombol = "Mulai Investigasi",
         bukan "Kasir Curang". Investigasi tetap dilakukan manusia.
         -> Buat apa: jawab Case D. Sistem mendeteksi pola yang tidak
         biasa dan memberikan sinyal ke pengurus untuk ditindaklanjuti.
         Bukan otomatis menuduh — menghindari fitnah kalau itu kesalahan
         teknis biasa.

[x] 8.3  Halaman Guard (/guard) — dua tab:
         Tab "Anomali": tabel per kasir + jumlah flag + waktu kejadian
           + status investigasi (belum ditinjau / sedang ditinjau / selesai)
         Tab "Audit Trail": filter tabel + rentang tanggal + nama user
           -> tampil tabel: waktu | siapa | aksi | kolom | nilai lama -> baru
         -> Buat apa: pengurus punya satu tempat untuk semua urusan
         integritas data. Audit trail juga dipakai saat pengawas datang.

[x] 8.4  Role pengawas eksternal (Case E):
         Login sebagai pengawas (dinas koperasi) -> bisa:
           SELECT audit_log koperasinya
         Tidak bisa:
           SELECT tabel anggota / ternak / pinjaman langsung
         RLS: audit_log WHERE koperasi_id = pengawas.koperasi_id
         -> Buat apa: pengawas bisa periksa "ada yang ubah data pinjaman
         dekat tanggal pencairan" TANPA mengakses data sensitif anggota
         yang tidak relevan. Privasi terjaga, transparansi tetap ada.

[x] 8.5  UJI Case D: seed data kasir dengan 5 void after-hours

[x] 8.6  UJI Case E: login sebagai pengawas -> filter tabel=pinjaman +
         tanggal pencairan -> lihat siapa ubah apa kapan ->
